## Component 1 — `shared/proof_runtime.py` + `shared/proof_pack.py` (truth gates)

**Verdict: ACCEPT-WITH-FIXES (P2 only)**

### What was checked and is genuinely solid

- **The SafePay rewiring is the single most important fail-open removal on this branch and it is correct.** The freeze-era `_best_x402_proof` machinery (scored candidate paths, trusted `artifact.get("duplicate_proof_rejected")` booleans, and a 402→200 status-code "handshake" treated as replay proof) is deleted wholesale. `build_safepay_lite` (proof_runtime.py:479) now renders exclusively from one strict registry item: exactly one `safepay_v2`/`temporal_scope=current` item (`_current_safepay_registry_item`, proof_runtime.py:457–476; `len(matches)==1` or `None` — two candidate items fail closed), gated by `proof_item_is_green`, and replay safety additionally requires the three named required checks (`provider_consumption_row_matches_payment_and_binding`, `exact_retry_returned_same_fulfillment_hash_without_second_consumption`, `cross_binding_reuse_returned_terminal_409`) each with `required is True and passed is True` (proof_runtime.py:494–506). Absence of the registry item renders `status="unverified"`, `payment_verified=False`, `included_in_governance_proof=False` — the historical June payment hash stays visible but is labeled historical. This matches §12/§13 exactly.
- **Caller injection into the invariant runner is closed.** `build_invariant_runner(evidence, safepay=None)` explicitly `del safepay` and reloads from the registry (proof_runtime.py:362–369). A success-shaped mapping passed by a caller can no longer flip `duplicate_x402_proof_rejected`. The invariant check (proof_runtime.py:410–414) is now derived, not asserted.
- `_policy_hash_guard_result` returns `passed: None, status: "missing_evidence"` when `policy_hash` is absent, and `build_invariant_runner` maps that to overall `"incomplete"` — absence is not success (proof_runtime.py:346–359, 433–440).
- The nonce-replay and tamper guards actually execute the production primitives (`shared.approval.validate_and_consume_nonce`, `compute_action_hash`) rather than reading stored results.
- `_allocation_from_prompt` input is bounded to 4096 chars with bounded digit quantifiers (linear-time regex; proof_runtime.py:626–640).

### Findings

**F1 (P2) — quorum invariant still trusts asserted artifact strings.**
`shared/proof_runtime.py:233–243` (`_quorum_blocking_proof_ok`): the `quorum_required` invariant passes iff `artifacts/live/odra-quorum-exercise-plan.json` says `status=="live_complete"`, `pre_quorum_execution_blocked.status=="verified"`, `final_receipt_after_quorum.status=="verified"`, and the artifact's deploy hash equals the pinned `CANONICAL_QUORUM_RECEIPT_HASH`.
*Failure scenario:* a hand-written JSON file at that path containing those four strings makes the "no quorum blocks execution" machine check render `passed: true` with zero observation — exactly the presence-equals-verified pattern this branch eliminated for SafePay. Mitigation in place: the deploy hash is pinned to a compile-time constant, and the file lives inside the deployment trust boundary; this is also **pre-freeze legacy behavior** (identical at `concordia-g1-freeze-v2.0-a`), not a regression. But the branch edited `build_invariant_runner` and set a registry-derivation standard it did not extend here.
*Minimal fix:* derive this check from the proof registry instead — the `exact_envelope_v3` item's `pre_quorum_finalize_reverted_with_code_8` required check (and/or the historical item) under `proof_item_is_green`, mirroring the SafePay pattern.

**F2 (P2) — audit-packet quorum/topology rows are asserted, not derived.**
`shared/proof_pack.py:563–577` (`load_odra_quorum_proof`) accepts artifact `status in {"live_complete","verified","complete"}` and `proof_pack.py:520–530` then appends a compact-proof-table row with hard-coded `"status": "verified"` and sets `locke_execution_firewall.on_chain_quorum_enforced = True` from artifact content alone. Same for `load_odra_topology_genesis_proof` (proof_pack.py:580–596; `status=="live_complete"` and module-name presence).
*Failure scenario:* fabricated artifact → public audit packet asserts "2-of-3 Odra quorum receipt executed — verified" and "on_chain_quorum_enforced: true" without any recomputation.
*Minimal fix:* same as F1 — status must come from the registry green predicate; the artifact may only supply display detail.

**F3 (P2) — absence-defaults-to-true on a stored display field.**
`shared/proof_pack.py:245`: `_format_stored_adversarial_attempt` sets `"live_gateway_validation": bool(attempt.get("live_gateway_validation", True))` — a missing field renders as `true` ("absence of a field treated as success" lens). The sibling fields default to `False`. Data source is the sealed evidence chain (internal), so exposure is a mislabeled display claim, not an authority path.
*Minimal fix:* default to `False`.

**F4 (P2) — inverted guard signal in the interactive replay.**
`shared/proof_runtime.py:665–667`: `unsafe = attempted > approved or approved_action_ok`, where `approved_action_ok` is `_action_hash_guard_rejects_tamper(...)` — i.e. "the two envelopes hash differently", which is true for *any* `attempted != approved`, including a below-cap request. A judge submitting "move 5%" against an 8% cap gets `status="blocked"`, `locke_result="refused_to_sign"` with `invariant_result="within_cap"` — internally contradictory output. This fails in the safe direction (never renders an over-cap attempt as allowed; never touches the chain: `casper_transaction_triggered: False`), but the hash-difference signal (proof the guard *works*) is being used as an unsafety signal.
*Minimal fix:* `unsafe = attempted > approved`; report the hash-mismatch separately as `envelope_binding_demonstrated`.

**F5 (P2) — RWA supplemental receipt status trusted from artifact.**
`shared/proof_runtime.py:719–732` (`build_rwa_evidence_run`): `proof.get("status") == "processed"` from `artifacts/live/dynamic-proposal-execution-proof-DAO-PROP-RWA-001.json` yields `supplemental_receipt_status: "processed"` plus an explorer link built from the artifact's `deploy_hash`. Asserted-string-equals-processed; clearly labeled supplemental, no canonical claim, hence P2.
*Minimal fix:* pin the expected deploy hash (constant already exists: `SUPPLEMENTAL_RWA_RECEIPT_HASH`) and require equality before rendering "processed", or route through the registry.

## Component 2 — Treasury executor hardening

**Verdict: ACCEPT — no findings. This is the strongest component on the branch.**

Files reviewed in full: `shared/treasury_executor.py` (1,510 lines), `shared/v3_authorization.py` (772), `scripts/run_treasury_execution.py` (operator wiring, signer loading, broadcast/reconcile callbacks), plus the supporting parsers' interfaces (`native_transfer_deploy`, `native_transfer_finality`, `native_transfer_scan`, `post_transfer_proof`, `casper_state_proof`, `treasury_snapshot`). Source analysis only; I could not execute the state machine, but every claim below is grounded in the exact code paths cited.

What was checked, adversarially, and held:

- **Exact-envelope binding.** `authorize()` accepts only a `VerifiedNativeAuthorization`, and `validate_verified_authorization` (v3_authorization.py:325–537) re-parses *every* embedded transcript from raw bytes on every use: canonical typed header/body JSON round-trip, `derive_native_material` recompute of `action_id`/`envelope_hash`/core bytes, two-observer snapshot artifact re-verification, readback artifact re-verification (`finalized=True`, `action_authorized=True`, quorum ≥ threshold), full v3 proof re-verification, and equality of all 13 identity fields against the recomputed material. The 32-byte `verification_seal` is an unkeyed integrity commitment, not the trust anchor — the re-verification is. A hand-constructed dataclass with forged fields fails `validate_verified_authorization` in `_row_to_entry` (treasury_executor.py:556) on every journal read, not just at insert.
- **Temporal/fork binding (G1-C6/C7/C8/C9).** snapshot height strictly `<` finalization height (v3_authorization.py:520–521); readback height `>=` finalization height, with equal-height forced onto the identical block hash and state root (v3_authorization.py:522–530); the no-duplicate scan must start at exactly the v3 finalization height **and** block hash (`hmac.compare_digest`, treasury_executor.py:724–730 on load and 1234–1240 in `prove_execution`) — a competing block at the authorization height is rejected. `amount_motes == floor(balance * bps / 10000)` is enforced structurally (treasury_executor.py:347–352).
- **Replay/double-submission.** Schema `PRIMARY KEY (network, action_id, envelope_hash)` + `UNIQUE (network, action_id)` (treasury_executor.py:260–261): one action can never bind two envelopes. Re-`authorize` with different immutable data → `JournalConflict` (829–839). Signed bytes are produced once, inside the `BEGIN IMMEDIATE` claim in `prepare` (deliberate, documented: signing is local; network I/O only in `broadcast`), committed with hash before any network write, revalidated byte-for-byte and hash-for-hash before every broadcast (993–1012), and *never rebuilt* — re-broadcast is chain-idempotent by identical deploy hash.
- **Crash/ambiguity windows.** `broadcast` commits `AMBIGUOUS_SUBMITTED` + inflight lease *before* invoking the callback (972–1030), so a crash mid-broadcast recovers through deploy-hash reconciliation only. Every callback exception maps to ambiguous. A broadcast-callback claim of `terminal_failure` is **not trusted** — it stays `AMBIGUOUS_SUBMITTED` with `unverified_broadcast_terminal_failure` (1055–1057). A reconcile-callback claim of `terminal_failure` likewise only records a detail code via `_retain_unverified` without a state transition (1308–1313). `finalized` claims must carry raw node observations that pass `verify_finalized_native_transfer` against the *persisted signed bytes* and expected source/recipient/amount/transfer-id/payment bounds (1314–1345); a mismatched deploy hash from the callback is retained-unverified, not recorded. Non-finalized status with attached finality evidence is rejected (`unexpected_finality_evidence`, 1295–1298).
- **State CAS discipline.** All transitions read-then-conditionally-UPDATE with `AND state=?` inside a single `BEGIN IMMEDIATE` transaction (`_write_transaction`, 477–489), so the unchecked `rowcount` is safe — no interleaving writer can exist between the read and the update, and cross-process races collapse onto the state checks (`_record_broadcast_result` refuses to touch anything no longer `AMBIGUOUS_SUBMITTED`, 1066–1067). WAL + `synchronous=FULL` + 30s busy timeout; `:memory:` refused (442–445).
- **Stored evidence is never trusted on reload.** `_row_to_entry` re-runs the full finality parser for FINALIZED/PROVEN rows and requires the stored scalar columns to equal the re-proven tuple (611–638); PROVEN rows additionally re-verify pre/post balance bundles at the exact snapshot/finality roots, re-verify the scan transcript, and recompute the execution-proof digest (642–747). Tampering with journal columns yields `JournalConflict`, not silent green.
- **Operator boundary.** Signer loaded via `read_secure_secret_file` with the account hash bound to the treasury source account; error text excludes path and contents (run_treasury_execution.py:315–356). `account_put_deploy` requires the explicit `allow_submit` flag on a transport that otherwise allowlists read-only methods and pins `https://…/rpc` (casper_rpc_transport.py:529–535). Broadcast treats *everything* after bytes leave the process as ambiguous (run_treasury_execution.py:766–774). Reconciliation demands per-endpoint status/deploy/block transcripts from all configured nodes with distinct origins (two-node corroboration enforced downstream by the finality verifier and the `successful_inclusion_observed_by_two_named_casper_rpc_nodes` check).

I attempted to construct: a double-broadcast via lease expiry (blocked by state CAS + identical bytes), a forged FINALIZED via callback (blocked — evidence re-parsed), a journal tamper to skip `prove_execution` inputs (blocked — digest + reparse on load), a same-action different-envelope re-authorization (blocked by UNIQUE + conflict), and a competing-block scan (blocked by the C9 hash equality). None succeeds in source.

## Component 3 — Proof registry (shared core + gateway routes + card-chain publication)

**Verdict: ACCEPT-WITH-FIXES (1 × P1, 2 × P2)**

### What was checked and is solid

- **§13 item shape is exact**: `PUBLIC_ITEM_REQUIRED_FIELDS` is precisely the 29-field set including `deployment_domain` (G1-C7) and `settlement_transaction`; per-proof-type required-check sets match §13's fixed lists (including `facilitator_verify_returned_is_valid_true` per G1-C11); provenance allowlists per proof type (`_PROVENANCE_BY_PROOF_TYPE`, proof_registry.py:159–223) implement G1-C11 relabelling protection; historical types cannot claim `current`, v3 types cannot claim `historical`, `snapshot` is generation-`none`/`not_applicable` only.
- **Green predicate is fail-closed** (`proof_item_is_green`, proof_registry.py:490–504): shape errors, non-verified status, `observation_mode=unavailable`, non-green outcomes, missing/demoted/failed required checks, duplicate check names, and any failed `required=true` extra check all deny green. Top-level asserted booleans have no path into it.
- **Chronology (G1-C12)**: strict RFC3339-UTC-Z regex; `check.observed_at > captured_at` → error; `captured_at > generated_at` → item invalidated; `generated_at > reference_time` → hard `ValueError` (proof_registry.py:434–441, 514–548).
- **Storage loader is strict**: duplicate-JSON-key rejection, 8 MiB cap, exact top-level field set, and the `card_chain_roots` identity pinned to `artifacts/live/card-chain-roots-v1.json` with a hex32 digest (`_strict_json`, proof_registry.py:657–682).
- **Internal endpoints conform to §13**: hex32 path validation, 404 on zero matches, 409 `ambiguous_governance_binding` on >1 (both lookups; `by_signed_payment_payload_hash` additionally filters to current-verified with derived `v3_finalized_exact` — proof_registry.py:735–753). `validate_internal_record` **derives** `v3_finalized_exact` and demotes `verified` when required checks don't all pass (619–645) — the stored boolean is overwritten, never trusted. Auth is `hmac.compare_digest` with empty-token fail-closed, checked **before** any lookup so 403 never discloses existence (gateway/app.py:555–559).
- **Public route** (`gateway/app.py:481–501`): known-proposal determination (canonical constant, registry, DB row, dynamic evidence) only chooses between 404 and an empty valid registry; loader failures are 503, not empty-success.
- **Card-chain publication (G1-C10) fails closed**: `build_card_chain_artifact` requires a 64-hex external release root — a missing/failed roots file yields `expected_final_card_hash=None` → `CardChainArtifactError` → 503 with `Cache-Control: no-store` (gateway/app.py:503–553; card_chain_artifact.py:544–548). Every card hash is recomputed from the exact stored preimage bytes, `previous_card_hash` linkage recomputed, wrapper/preimage identity cross-checked, `card_hash` forbidden inside the preimage, proposal identity bound per frozen card-type schema, secret scan applied to the exact preimage, reads bounded (per-card and cumulative byte caps, count cap, SQLite length limit) inside one deferred transaction, and the chain is cut exactly at the externally pinned receipt root (card_chain_artifact.py:380–590). Asserted `chain_valid` booleans have no input path.
- **Agent-identity work**: the `GATEWAY_SECRET` full-ACL fallback is removed from both auth loaders (gateway/auth.py, gateway/routes/submission.py) per §12 room-identity; key/role/agent-id maps are validated one-to-one at startup (`validate_agent_identity_configuration` in lifespan) and duplicates abort boot; `ProposalRoomClient` no longer transmits caller-supplied `sender_id`/`sender_role`/`sender_type` (shared/proposal_room.py:85–97) — identity is derived server-side.

### Findings

**F6 (P1) — public path serves `verification_status="verified"` even when a required check has `passed=false`; internal path demotes, public path does not.**
`shared/proof_registry.py`: `_public_item_errors` (372–487) validates that every required check is *present* with `required=true`, but never inspects `passed`. Consequently `normalize_proof_item` (507–511) and `build_public_registry` (514–548) leave `verification_status="verified"` intact on an item whose required check carries `passed=false`, and `GET /proof-registry/v1/{proposal_id}` serves it verbatim. The internal path has the correct behavior: `validate_internal_record` (637–639) demotes `verified → invalid` under exactly this condition. This is a real (if narrow) stored-boolean-over-derived-evidence sliver on the §13 public transport itself.
*Failure scenario:* a registry document written into `artifacts/live/proof-registry/` (assembler bug, manual edit, or any write-capable compromise of the artifacts volume) contains a `safepay_v2` item with all 11 required checks present, `required=true`, one `passed=false`, and `verification_status="verified"`. The public API — the §13 contract surface external verifiers and the dashboard lane consume — asserts `"verification_status": "verified"`. Green-cue consumers are protected (`proof_item_is_green` returns `False`; the npm verifier independently reports the item non-verified and would flag the whole registry), so no in-repo surface renders green — but the transport itself makes a claim the evidence contradicts, and any third party trusting the served field is misled.
*Minimal fix:* in `normalize_proof_item` (or `build_public_registry`'s per-item loop), demote `verification_status` to `"invalid"` whenever the item claims `verified` but the required-check pass predicate fails — i.e. reuse the exact check-evaluation block of `proof_item_is_green` (lines 499–504) as a demotion condition, mirroring `validate_internal_record`. ~6 lines; no schema change.

**F7 (P2) — internal-registry service token accepts a direct env value; spec freezes file-only loading.**
`gateway/app.py:556`: `_proof_registry_service_authorized` uses `read_secret("X402_GATEWAY_TOKEN")`, and `shared/runtime_secrets.read_secret` prefers a plain `X402_GATEWAY_TOKEN` env value over `X402_GATEWAY_TOKEN_FILE`. §13 states the token "comes only from `/run/secrets/x402_official_gateway_token`", and §12 (approval boundary) sets the production norm that direct value variables are ignored. A direct-env token weakens nothing cryptographically, but it reintroduces the leak surface (process env is visible to `docker inspect`, crash dumps, spawned children) the `_FILE` convention exists to close, and it deviates from the frozen contract.
*Minimal fix:* load this specific token file-only (require `X402_GATEWAY_TOKEN_FILE`, ignore the direct variable), or add a `read_secret_file_only` helper and use it for every §12/§13-frozen secret.

**F8 (P2) — cross-language strictness asymmetries in the public document (single consolidated finding).**
(a) Duplicate `proof_id`s across merged stored documents are not rejected by the Python public builder, while the TS verifier hard-fails the entire registry on them (`registry.ts:324–331`) — a Python-served document can be structurally un-verifiable by the published tool. (b) Check-name length ceilings differ: Python `_CHECK_NAME_RE` allows 128 chars (proof_registry.py:228), TS `CHECK_NAME` allows 96 (registry.ts:288). All frozen names (max 79) pass both; only hypothetical future names diverge. Both asymmetries fail closed on the npm side, so this is conformance hygiene: mirror the duplicate-`proof_id` rejection into `build_public_registry` and align the two regex ceilings.

## Component 5 — npm verifier `@concordia-dao/verify`

**Verdict: ACCEPT (1 × P2 conformance divergence)**

Reviewed: `src/registry.ts` (997 lines, in full), `src/cli.ts`, `src/modes/local.ts`, adapter surfaces (`casper-deploy-json.ts` signature handling, adapter fact plumbing), `package.json`, `README.md`.

- **It verifies what it claims.** README claims were cross-checked against source: deploy-approval signatures *are* verified (node:crypto — Ed25519 raw, secp256k1 SHA-256/P1363, `casper-deploy-json.ts:197–201, 550–556`; no curve dependency needed, sole runtime dep is `@noble/hashes` pinned at 1.8.0); deploy and body hashes recomputed via BLAKE2b-256; forged/asserted booleans (`passed`, `chain_valid`, `verified`, `duplicate_proof_rejected`, …) are quarantined into `ignoredAssertions`, never consumed.
- **Fail-closed on missing fields.** All 29 item fields must be *own properties* (registry.ts:391–393); unknown item and top-level fields are invalid; `verified` without loadable artifact bytes is `unavailable`, never verified (473–486, 582–586); `verified` with bytes requires the strict per-type adapter to independently re-derive the claims, and every adapter-derived identity field must equal the item's claim (`validateAdapterClaims`); proof types without a shipped adapter (safepay_v2, approval/demo/room, official x402, historical v2) are *documented* as remaining `unavailable` even when their checks assert success — the honest opposite of vacuous pass. Status precedence is invalid > unavailable > unknown > verified; nothing promotes silently (589–603).
- **Chronology & provenance:** future `generated_at`/`captured_at` vs. verifier reference time rejected (G1-C12); `VERIFIED_PROOF_SEMANTICS` enforces the G1-C11 provenance allowlists; duplicate `proof_id` and duplicate check names are registry-fatal; 128-item cap.
- **Cross-proof binding (G1-C7/C9):** a verified treasury item requires exactly one identity-matched, independently verified exact-envelope item, with `snapshot < finalization`, `authorization scan-start == finalization` (height *and* hash), else the treasury item is invalidated (651–697).
- **I/O hardening:** local mode `O_NOFOLLOW` + realpath confinement of artifacts to the registry directory, per-file and aggregate byte caps, bounded chunked reads verifying the opened file identity; strict JSON with duplicate-key rejection; url/live modes HTTPS-only, redirect-refusing, address-pinned, credential-rejecting, with a read-only RPC allowlist (per README and `modes/` source). CLI exit codes 0/2/3/4/64 are distinct and never conflate unavailable with verified.

**F9 (P2) — `<` vs `<=` divergence on native-execution ordering.**
`registry.ts:689`: `exact.finalizationBlockHeight >= facts.nativeBlockHeight` invalidates the treasury item — i.e. the native transfer must be in a *strictly later* block than the v3 finalization.
